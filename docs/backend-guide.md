# nanhai-data-map 后端说明文档

## 1. 后端的作用是什么

这个后端是“南海区数据产业企业图谱”项目的数据服务层，核心作用是把企业信息统一保存到数据库里，并通过 HTTP 接口提供给前端或其他程序使用。

它现在主要解决的是两个问题：

1. 把企业图谱数据结构化保存下来
2. 让前端或脚本可以按统一格式读取和新增企业数据

结合项目里的 [taxonomy.md](./taxonomy.md)，这个后端实际上是在承接“企业分类结果落库”这件事。也就是说，前面无论是人工整理、爬取、还是智能体分类，最终都可以把结果存成一条条企业记录。

---

## 2. 当前后端包含哪些模块

### `backend/main.py`

这是后端入口文件，作用有三个：

1. 创建 FastAPI 应用
2. 启动时自动创建数据库表
3. 注册企业接口路由

它相当于整个后端的启动总控。

### `backend/database.py`

这个文件负责数据库连接配置，目前使用的是 SQLite。

它做了三件事：

1. 定义数据库地址 `sqlite:///./enterprise.db`
2. 创建 SQLAlchemy 引擎和会话工厂
3. 提供 `get_db()`，让每个接口都能安全地拿到数据库连接

### `backend/models.py`

这个文件定义数据库表结构，也就是企业数据在数据库里到底长什么样。

当前只有一张表：`enterprises`

主要字段包括：

- `name`：企业名称
- `town`：所在镇街
- `category`：企业分类
- `category_reason`：分类依据
- `products`：主营产品
- `source_url`：来源链接
- `evidence`：证据片段
- `confidence`：置信度
- `reviewed`：是否人工复核

这张表和 `taxonomy.md` 的关系很紧密，其中 `category` 字段就是分类体系落地后的结果。

### `backend/schemas.py`

这个文件定义接口收发数据时的格式。

它不是数据库表，而是“接口层的数据规则”，作用是：

1. 校验前端传来的数据是否合法
2. 规定接口返回的数据结构
3. 自动生成 FastAPI 的接口文档

当前有三个模型：

- `EnterpriseBase`：企业公共字段
- `EnterpriseCreate`：创建企业时使用
- `EnterpriseResponse`：返回给前端时使用，额外带 `id`

### `backend/api/enterprise.py`

这个文件是企业接口模块，目前有两个接口：

1. `POST /enterprises/`
作用：新增企业记录

2. `GET /enterprises/`
作用：获取企业列表，可按 `category` 分类筛选

这说明当前后端还是“基础版 CRUD 后端”的早期形态，重点在于先把数据存起来和查出来。

## 3. 这个后端的整体工作流程

整个流程可以理解成下面这样：

1. 启动后端服务
2. FastAPI 加载 `main.py`
3. `main.py` 调用 SQLAlchemy，根据 `models.py` 自动建表
4. 前端或脚本向 `/enterprises/` 发送请求
5. FastAPI 用 `schemas.py` 校验数据
6. `enterprise.py` 调用数据库会话，把数据写入或读取出来
7. 数据最终保存在项目根目录的 `enterprise.db` 中

简化理解就是：

“接口层接请求，Schema 做校验，Model 定结构，Database 管连接，SQLite 真正存数据。”

---

## 4. 目前这个后端能做什么

当前版本已经支持：

1. 启动一个可访问的 API 服务
2. 自动创建本地数据库文件和企业表
3. 新增企业数据
4. 查询所有企业数据
5. 按 `category` 字段筛选企业

所以它已经能支撑一个最基础的“企业图谱数据录入 + 查询”场景。

---

## 5. 目前这个后端还不能做什么

从代码来看，它现在还比较简化，暂时没有这些能力：

1. 没有删除和修改接口
2. 没有分页
3. 没有按企业名称搜索
4. 没有多条件筛选，比如按镇街、是否复核筛选
5. 没有用户登录和权限控制
6. 没有数据去重逻辑
7. 没有批量导入接口
8. 没有把 taxonomy 的分类规则做成自动判定逻辑

所以它更像是项目的“第一版后端底座”。

## 6. 怎么启动这个后端

先进入项目根目录：

```bash
cd /mnt/d/mpb/nanhai-data-map
```

如果你本地还没装依赖，可以先安装常用依赖：

```bash
pip install fastapi uvicorn sqlalchemy pydantic
```

然后启动服务：

```bash
uvicorn backend.main:app --reload
```

启动成功后，默认访问地址一般是：

```text
http://127.0.0.1:8000
```

---

## 7. 怎么确认服务是否正常

浏览器打开：

```text
http://127.0.0.1:8000/
```

如果返回：

```json
{"message":"Nanhai Data Map API is running"}
```

说明后端已经启动成功。

你也可以打开自动接口文档：

```text
http://127.0.0.1:8000/docs
```

这里会看到 Swagger UI，可以直接测试接口。

## 8. 怎么使用接口

### 8.1 新增企业

请求地址：

```text
POST /enterprises/
```

请求示例：

```json
{
  "name": "示例数据科技有限公司",
  "town": "桂城街道",
  "category": "数据技术类",
  "category_reason": "官网明确介绍其提供数据分析平台和机器学习解决方案",
  "products": "数据分析平台、AI建模系统",
  "source_url": "https://example.com",
  "evidence": "提供大数据分析、智能建模和企业数据中台服务",
  "confidence": 0.92,
  "reviewed": false
}
```

返回结果会包含数据库自动生成的 `id`。

### 8.2 查询企业列表

请求地址：

```text
GET /enterprises/
```

作用：获取全部企业列表。

### 8.3 按分类筛选企业

请求地址：

```text
GET /enterprises/?category=数据技术类
```

作用：只返回某个分类下的企业。

## 9. 数据库文件在哪里

当前数据库使用 SQLite，本地数据库文件是：

```text
enterprise.db
```

它在项目根目录下，也就是：

```text
/mnt/d/mpb/nanhai-data-map/enterprise.db
```

这意味着这个项目目前非常适合本地开发、课堂展示、原型验证和小规模数据管理。

## 10. taxonomy 文档和后端是什么关系

`docs/taxonomy.md` 提供的是“企业怎么分类”的规则。

后端负责的是“把分类结果存下来并提供查询接口”。

两者关系可以理解为：

- `taxonomy.md` 是分类标准
- `backend` 是分类结果的存储和服务系统

例如：

1. 你先根据 `taxonomy.md` 判断某企业属于“数据技术类”
2. 再把这个分类结果写入 `category`
3. 把判断理由写入 `category_reason`
4. 把证据来源写入 `evidence` 和 `source_url`

这样一条企业记录就同时具备了“分类结果”和“可追溯依据”。

## 11. 适合怎么继续扩展

如果你后面要继续完善这个项目，建议优先做下面几项：

1. 增加 `PUT` 和 `DELETE` 接口，支持修改和删除企业
2. 增加按名称、镇街、是否复核的筛选
3. 增加分页，避免企业太多时一次返回全部数据
4. 增加批量导入 CSV 或 Excel 的能力
5. 增加去重规则，避免重复录入企业
6. 增加辅助标签字段，对应 `taxonomy.md` 里的建议标签
7. 把分类逻辑做成半自动或自动分类接口

---

## 12. 一句话总结

这个后端本质上是一个面向“南海区数据产业企业图谱”的轻量级数据管理服务。

它当前负责把企业分类结果、分类依据和证据材料保存到数据库，并通过接口提供新增和查询能力，方便前端展示、人工复核和后续数据分析使用。
